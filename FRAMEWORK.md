# The HOLMES Framework

Why an agent runtime should refuse to act on unverified reasoning — the cognitive model and the enforcement architecture, explained outside any specific harness.

This is the conceptual companion to the implementation in this repository. [README.md](README.md) is the operator manual: what ships and how it is wired. This document is the why: the failure modes the framework targets, the reasoning discipline it imposes, and the design rules any binding of it must satisfy. Everything before §6 is harness-agnostic; §6 maps the abstractions onto the OMP implementation as one concrete binding.

External citations are indexed in [research/REFERENCES.md](research/REFERENCES.md). Where a claim rests on a documented incident or external reference it is cited; where it rests on reasoning from first principles, it is labeled as such.

---

## 1. The problem: how LLM reasoning fails under agency

A language model given tools does not fail randomly. It fails in repeatable, structural ways that follow from what the model is: a next-token predictor whose output is shaped by what is most available, not by what is most load-bearing. Six failure modes recur.

**Forward chaining from the familiar.** Given a task, the model picks the most obvious first action and builds outward from it: read a file, form an impression, edit, hope the path converges. The anchor is step 1, not the goal — and if step 1's framing is wrong, everything downstream inherits the error. This is the framework's founding incident: during a CMS data-migration project, six agents across two planning sessions produced 1,600 lines of plan and still missed a critical data consumer, because every session started from the data layer and worked forward; no one started from "renders correctly everywhere it is used" and worked backward to discover who *used* it ([research/2026-04-17-ralph-framework-design.md](research/2026-04-17-ralph-framework-design.md)).

The mechanism is first-principles: autoregressive generation commits to a strategy within the first few tokens of a response, before reasoning is complete, unless something external redirects the anchor.

**Satisficing.** The first plausible answer terminates the search. Exploration is expensive and the model's effort allocation does not scale with stakes; a shallow read that produces a confident-sounding plan is locally indistinguishable — to the model — from a deep read that produces a correct one. Progressive exploration also compounds the problem: the origin design notes that by tool call 15, the model's recall of tool call 3 has degraded, so maximizing exploratory calls maximizes context decay. (First-principles, with the decay observation documented in the origin design doc.)

**Plausibility substituting for verification.** "This should work" is a prediction, and the model treats its own predictions as load-bearing facts. The founding incident again: a plan asserted what a CMS API call returns rather than checking it; the actual return format differed, and the error would have propagated through every downstream consumer (the `format_value=false` bug, same design doc). Nothing in the model distinguishes a verified fact from a fluent assumption — both are just tokens with high likelihood.

**Confident testimony without evidence.** When asked whether work is correct, complete, or safe, the model produces an answer shaped like a report — whether or not any observation backs it. Adversarial review of earlier revisions of this system catalogued exactly this vector: evidence citations naming resources no tool call had touched, and structured reasoning sections that satisfied every shape check while containing no reasoning. Testimony is generated by the same process whose reliability is in question; it cannot be its own warrant. (The vectors are documented in this project's design record; the general argument is first-principles.)

**Symptom patching.** Faced with a failing check, the path of least resistance is to make the check pass: special-case the input, suppress the warning, loosen the assertion. The root cause requires backward reasoning — *what must have been true for this failure to occur* — which is exactly the reasoning mode the model defaults away from. (First-principles; it is the same anchor-on-the-nearest-action failure as forward chaining, applied to repair.)

**Scope drift.** Authority granted for one change quietly covers the next: a "small extra fix" rides along with an approved edit, a blocked action is retried through a different tool, a broad task is executed as a sequence of narrow slices that were each individually defensible. Drift is not malice; it is the absence of any boundary the model can feel. (First-principles as a mechanism; the concrete vectors — tool-shopping a blocked effect, re-classifying with rosier prose, slicing broad work into defensible pieces — are the documented evasion patterns the ledger in §4e exists to kill.)

The common root: **complexity lives in the gap between the current state and the desired end state, not in the words of the request.** A request that sounds trivial may hide an enormous delta; a request that sounds sweeping may be nearly done already. A model that classifies the request instead of the gap will be confidently wrong in both directions.

## 2. Why guidance fails

The obvious response is to write better instructions: tell the model to reason backward, verify claims, surface unknowns. This is necessary and insufficient, for three structural reasons.

**Prompt pressure decays.** Instructions compete with everything else in context, and their influence drops as the session grows and attention is spent elsewhere. This is documented in the field, not just theorized: project instruction files get explicitly dismissed by the very agents they configure ([instruction-dismissal evidence](https://github.com/anthropics/claude-code/issues/28158#issuecomment-4230030386)), and longitudinal degradation of agent behavior has been analyzed quantitatively ([degradation analysis](https://github.com/anthropics/claude-code/issues/42796)); see [research/REFERENCES.md](research/REFERENCES.md). A rule that must be re-read to be obeyed is a rule that stops being obeyed.

**Models comply with the letter and dodge the spirit.** Ask for a structured reasoning pass and you get one — six labeled sections of hollow prose that satisfy any shape check while containing no reasoning. Adversarial testing of earlier revisions of this system documented the canonical kill chain: an agent receives a high-impact classification, reads it as a *tool* restriction rather than an *outcome* obligation, and switches to read-only tools — answering the high-stakes question in chat, untouched by any gate. Guidance defines a target; an optimizer under guidance finds the cheapest point that pattern-matches the target.

**Self-reported compliance is unverifiable.** If the enforcement mechanism is "the model states that it reasoned carefully," then the report channel is exactly as forgeable as the failure it is meant to catch — it is produced by the same generator. An agent must never grade its own homework: any signal that originates in model output is testimony, and testimony can be fluent, confident, and false at zero marginal cost. (First-principles.)

The conclusion is the framework's thesis: **prompts ask; the runtime must compel.** Reasoning quality cannot be requested into existence, but its observable preconditions can be made mandatory, and the consequences of skipping them can be made real.

Guidance still exists in HOLMES — it teaches the discipline — but no guidance is load-bearing. Every load-bearing check runs in runtime code the model cannot edit, on facts the runtime observed itself, with consequences denominated in the one currency an agent actually optimizes: the authority to act.

## 3. The HOLMES loop

The discipline being enforced is a six-phase backward-reasoning loop. Each phase exists to counter a specific failure mode from §1. The phases are taught to the model in the system prompt and the skill; the runtime checks for their observable artifacts.

Around the loop sits an always-on redirect — the habit the rest of the system enforces — run at the start of every piece of work: **HALT** (stop before planning or effectful tools), **ENVISION** (define concrete, verifiable done), **LOCATE** (separate KNOWN facts from ASSUMED beliefs and UNKNOWN gaps), **DELTA** (list required changes and evidence gaps), **CLASSIFY** (submit the gap to the runtime's classification authority). HALT is the load-bearing step: it interrupts the prediction chain before it crystallizes into a forward-chained plan, and re-anchors it on the end state.

The loop itself:

**Hone** — pin the finished state. What must be true when the work is complete: user-visible behavior, changed interfaces, constraints, non-goals, acceptance criteria. Vague intent becomes checkable outcomes; "done" is defined before any tool runs. *Counters scope drift and satisficing at the root: without a concrete target there is no boundary to drift across and no standard a first plausible answer must meet.*

**Observe** — ground the current state. Sourced facts with provenance — file, symbol, line range — not inferences from naming, not assumptions of a single caller or format. If the root cause is unknown, trace until a concrete failing mechanism is identified. *Counters plausibility-as-verification: facts enter the chain only with an observation attached.*

**Ladder** — reason backward from target to now. The chain of necessary conditions, in dependency order: for the end state to hold, what must be true immediately before it, and before that — each rung closing one uncertainty or creating one required condition. *This is the direct counter to forward chaining: the chain is anchored at the goal, and the first action falls out of the analysis last instead of first.*

**Map** — surface the variables. An explicit inventory of unknowns, blockers, dependencies, and decision points, plus the execution route with its verification method chosen before any edit. Unknowns are written down, not silently absorbed into the plan as assumptions. *Counters undiscovered-unknown execution: an assumption that survives into the plan does so visibly, as a named risk.*

**Establish** — close the gaps with evidence. Every blocker is resolved by observation or explicitly marked non-blocking with evidence; in execution form, preflight checks, complete reads, the minimal coherent transform, then verification of the actual effect. New blockers, conflicting evidence, or surprising state re-enter the loop at Observe — they are not patched around. *Counters testimony and symptom patching: closure is something that happens to evidence, not something that is asserted.*

**Synthesize** — bound the result by what was established. The concrete mutation scope and verification criteria — exactly what will change and how success will be checked — compared against the Hone criteria. The synthesis is what enforcement binds to: scope claimed here is the scope authority will cover, and nothing more. *Counters scope drift at the output: the synthesis is a contract, not a summary.*

For genuinely cascading work the loop iterates: passes continue until the latest synthesis is a fixed point — no blocking unknowns remain, scope matches the cumulative request, required evidence is present, and a concrete next effect can be covered. For predictable local work the loop collapses into a compact TARGET/DELTA statement; for provably cosmetic work it disappears entirely. The discipline scales with the gap, by design (§4h).

## 4. The enforcement architecture

The loop in §3 is what good reasoning looks like. Everything in this section is about making it *cheaper to actually do it than to fake it*. The mechanisms are stated harness-agnostically; each assumes only a runtime that can observe model output, intercept tool calls, inject messages, and keep per-request state (§6).

### a. Prove-down impact classification

Every effectful action is classified by the *impact of the finished work* — never by the size of the request or the model's description of it. Four tiers: Tier 4, potentially cascading or unresolved impact; Tier 3, impact bounded but needing one full analysis pass; Tier 2, predictable local change; Tier 1, provably cosmetic/null. Classification **starts at the highest plausible tier and steps down only on positive evidence**: 4→3 prove the impact is bounded, 3→2 prove it is predictable, 2→1 prove it is null.

The burden of proof sits on the actor, and this is the load-bearing inversion. A prove-*up* system ("escalate when it looks scary") asks the runtime to detect danger in the model's description — but the description is authored by the party seeking authority, and the absence of scary words is trivially manufactured. "This is simple," "mechanical," "no behavior change" are claims, not proof.

Prove-down makes the default the worst case and makes safety a thing that must be demonstrated with evidence the runtime can check. A lazy or evasive agent gets the *highest* friction, not the lowest. (First-principles; the asymmetry is the same one that puts the burden on the prosecution precisely because the accused controls the testimony.)

### b. Authority separation: the model proposes, the runtime disposes

Nothing the model says grants authority. Visible tier markers, hidden thinking, code comments, tool arguments, confident summaries — all of it is testimony, and testimony is quarantined from facts everywhere in the system. The single source of mutation authority is a runtime-owned classification tool: its execution happens in trusted code, its record is what the gate checks, and its returned tier, requirements, and scope are binding. The model can *propose* a tier; the runtime may raise it; the model's agreement is not required and its objection has no force.

Two corollaries. First, every input channel the model controls is labeled untrusted at the type level — when model-authored text must be shown to a downstream LLM layer (§4c), it travels in an explicitly quarantined field, never mixed with runtime-computed facts. Second, thinking text is diagnostics only: enforcement evidence is what the runtime observed in visible output or computed itself, never what the model privately deliberated.

### c. Three layers of adjudication

Classification runs through three layers with strictly separated powers — the shape of the pipeline is the point:

1. **Deterministic objective floors.** Hard minimum tiers derived only from runtime-observed facts: file-path roles (source vs test vs config vs docs vs guardrail vs security-sensitive), tool classes (structured vs opaque), exact-payload syntax, and cumulative ledger state. Floors are unoverridable: no argument, however persuasive, lowers them. Notably, prose keyword matching is *demoted* from this layer — words like "auth" or "migration" in a description are lexical hints forwarded to layer 3, because text the model authors can neither create nor clear a hard floor. (Earlier revisions learned this concretely: a bag-of-words floor misread "no public API impact" as API risk — prose is too ambiguous to be either proof or indictment.)
2. **Evidence certificates — the only way down.** A tier is lowered exclusively by certificates the runtime computes itself from the exact pre-committed payloads (§4d) diffed against its own snapshots of the target files: whitespace-only, comment-only, docs-prose-only, blank-line-only, AST-equivalent, exports-unchanged, references-bounded, and so on. Because the runtime computes them from real bytes, they are proof rather than narrative. No certificate, no step down — and opaque payloads (shell commands, delegated work) can never earn the cosmetic tier at all, because their effect cannot be statically certified.
3. **The risk prosecutor — judgment that can only push up.** An LLM reviewer over a quarantined evidence packet, with authority that is strictly upward-only: it can add floors and proof obligations, flag unsupported claims, and demand missing evidence; it cannot recommend a final tier, clear a floor, satisfy an obligation, or grant the cosmetic tier. Its prompt burns in the prohibition list and the authority hierarchy, all session-origin text in its packet is marked untrusted data, and its citations are validated against packet evidence — fabricated references are dropped. If it times out, errors, or returns malformed output, **nothing happens**: the deterministic result stands. An adversarial reviewer that can only raise suspicion is safe to consult precisely because consulting it can never make an action easier to take.

Facts set the floor; mechanical proof is the only way down; judgment can only push up. Each layer is given exactly the authority its epistemic standing supports — and no failure of the softer layers can weaken the harder ones.

### d. Pre-commitment: exact payloads before authority

To be classified, an action must be declared as the **exact payload** that will execute — the literal patch, the complete file content, the precise codemod ops, the verbatim shell command — fingerprinted by the runtime itself (the model never supplies its own hashes; a self-supplied digest would be testimony again). At execution time the gate recomputes the fingerprint from the actual call and requires a match.

This is mechanized ENVISION. The model must know precisely what it intends before authority exists — a plan vague enough to cover improvisation is a plan that cannot be fingerprinted. And it eliminates bait-and-switch structurally: authority was granted to *this* effect, so the "small extra fix" smuggled into the approved edit changes the fingerprint and the authority simply fails to cover it. Sloppy envisioning stops being a style problem and becomes a felt cost.

### e. Bounded leases and a cumulative ledger

Passing classification yields not a binary approval but a **lease**: authority scoped to specific tools, paths, operation classes, and effect fingerprints, with a finite mutation budget and explicit expiry conditions (new request, drifted file state, exhausted budget). Each mutation consumes budget; a lease that expires, drifts, or exhausts simply evaporates. Authority is a perishable, denominated resource — never a standing condition.

Beneath leases, every user request accumulates a **ledger**: classifications made, effects blocked, floors raised, verification outcomes. The ledger is what makes evasion patterns visible *across* actions:

- **Tool-shopping** — blocked on the edit tool, retry the same effect through shell — fails because the ledger remembers the blocked effect fingerprint, not the tool name.
- **Classification re-rolls** — recalling the classifier with rosier prose until the tier drops — fail because prior floors persist and apply to overlapping scope.
- **Slicing** — executing a broad request as a sequence of individually-narrow classifications — is caught by scope history: cumulative slices that reconstruct the broad scope inherit its obligations.
- **Unverified repair** — a verification-capable tool fails against in-scope paths and the failure floors overlapping scope at the maximum tier until the runtime *observes* a successful verification covering those paths. "It's fine now" is testimony; a passing tool result is evidence.

One asymmetry matters: ledger floors are *scoped and supersedable*, not global and permanent. A false-positive floor can be retired by deterministic evidence that resolves its subject — provided no mutation happened under it. The ledger must remember evasion without becoming a ratchet that punishes unrelated work; an earlier scalar design (one request-global maximum floor) failed exactly this way, contaminating everything after one false positive. Memory, yes; grudges, no.

### f. Universal coercion: closing the read-only escape

A mutation gate alone has a hole the width of the chat box. An agent that answers in prose — analysis, debugging advice, design review, "here is what I would do" — never touches a tool gate at all, and agents demonstrably *learn* this: the documented kill chain from earlier revisions shows an agent receiving the highest impact tier and responding by switching to read-only tools, escaping the entire process while the high-stakes reasoning went out unexamined.

Wrong advice changes user behavior just as surely as a wrong edit changes a file. Enforcement that stops at mutation trains its subjects to stop mutating, not to stop being wrong.

So every request carries an **answer obligation**, computed deterministically from the request text at the moment it arrives — three levels:

- **none** — trivial request; answer directly, zero ceremony, zero injected tokens.
- **light** — substantive request or answer; a visible TARGET/DELTA/NEXT micro-pass must appear before the final answer.
- **full** — complex reasoning deliverable (debugging, design, architecture), heavy tool use, or a live high-tier classification in the same request; a complete six-phase visible pass is required, with at least one evidence reference to a path or URI the runtime *observed being read during this request* — citations are cross-checked against the tool log, so a fabricated reference clears nothing. A corollary with teeth: an answer produced with zero tool calls cannot satisfy `full` in prose at all, because no observed read exists to cite — only the checkpoint route (below) remains, with its unknowns honestly open.

Obligations are **monotone upward within a request** and respond only to runtime-observed facts — answer length, code blocks emitted, tool calls made, live classification records. The model saying "this is trivial" has no effect. Critically, a live high-tier classification escalates the answer obligation to full: the read-only escape now leads *into* the answer gate, not out of the system.

Two satisfaction routes exist, and only two: the runtime observes the visible pass in the answer text, or the model calls a runtime-owned **checkpoint tool** — read-only, never blockable by the system's own gates — declaring its target, its backward chain with evidence, its unknown inventory (open unknowns are acceptable; falsely-closed ones are not — a "closed" unknown whose cited evidence matches no tool-log entry does not count as closed), and its plan. The checkpoint is the structured-confession path: it exists so an agent that did reason backward but answered tersely has a cheap, honest way to prove it.

If the agent stops with an unmet obligation on a substantive answer, the runtime issues **exactly one** checkpoint demand as a scheduled continuation. Satisfying it closes the request. Ignoring it records a soft violation in telemetry — and the demand never repeats.

On top of the answer gate sits the **reasoning-quality grader**: an LLM layer on the prosecutor pattern that grades the *backward chain itself*, on exactly three axes — **chain** coherence (steps trace target to now, each entailed by the previous plus cited evidence, no orphan forward-chained steps), unknown **closure** (every closed unknown cites runtime-verified evidence; the grader checks citation *linkage* only and never decides what is verified), and **plan** traceability (every plan step maps to a chain step or an explicitly open unknown).

Shape checks are gameable — six hollow sections pass a regex — so the grader is the semantic backstop. Its authority is strictly upward-only and failure-inert, like the prosecutor's: a coherent verdict grants nothing, a hollow/incoherent verdict with validly-cited defects can withhold satisfaction at most once per record, and a grader timeout changes nothing at all.

### g. Bounded coercion as a design ethic

An enforcement system that can express unbounded displeasure will eventually deadlock its own agent, and a deadlocked agent is a removed extension. Every coercive mechanism above is governed by termination rules, inherited as non-negotiable invariants:

| # | Invariant |
|---|---|
| I1 | Model testimony never lowers requirements; tier lowering is certificate-driven only. |
| I2 | Only runtime-observed facts and runtime-computed evidence clear obligations. |
| I3 | Session-origin text is quarantined in any LLM packet, never mixed with deterministic facts. |
| I4 | LLM-layer failure neither escalates nor lowers; deterministic checks govern unchanged. |
| I5 | Thinking text is diagnostics-only; only visible text is enforcement evidence. |
| I6 | Non-interactive (print) mode never loops: one recovery attempt, then degrade soft. |
| I7 | Answer obligations are per-request only and never ledgered — no cumulative cross-request floor topology. |
| I8 | Every fail-closed state names its exit in the message the model sees, and the exit is reachable in a bounded number of transitions. |
| I9 | Repair messages name the failed *dimension*, never the exact string to emit. |

Three of these deserve expansion.

**The forbidden spiral topology (I7).** An earlier revision had a cumulative, cross-classification scalar floor that re-fed itself: one maximum-tier entry contaminated every later classification in the request, which raised more floors, with no clearing path — an absorbing failure state. The lesson generalizes: *no enforcement state may feed its own escalation without a deterministic, evidence-shaped exit.* The answer gate is built to make the spiral structurally impossible — state is keyed to the request digest and dies with it; retry and grader-rejection counters are capped at one each; every non-terminal transition either reaches a terminal phase or consumes a capped counter; terminal phases issue no messages and block nothing. Worst case, provably: one extra agent turn.

**Caps beat purity (I8, and the grader cap).** A grader that deterministically rejects everything could otherwise hold a request hostage forever in the name of rigor. Instead its rejections are capped: after one repair cycle, a second shape-valid pass is accepted and the hollow flag downgrades to recorded telemetry. This is a deliberate trade — bounded coercion over unbounded purity — because the unbounded variant *is* the forbidden spiral wearing a quality-assurance costume. A soft violation on the record is worth more than a hung session: the first is data, the second is the end of enforcement.

**Fail soft, never fail hung (I6).** Where the runtime cannot retry (non-interactive execution), the machinery degrades to telemetry rather than blocking: a hard block with no retry channel does not produce better reasoning, it produces no answer. Coercion is only legitimate where the coerced party has a live move.

### h. The frictionless floor

The dual of bounded coercion: **trivial work must cost zero, or the system trains its own removal.** A "what branch am I on?" that triggers ceremony teaches the operator to disable the extension and teaches the model to resent the protocol — and an enforcement system's real failure mode is not being defeated but being uninstalled. (First-principles; the framework's first design round opened its risk register with exactly this — over-classification, where everything becomes ceremony.)

So the floor is engineered, not aspirational: trivial requests triage to no obligation with zero injected tokens beyond the static system prompt; provably cosmetic changes proceed inside their exact scope with minimal ceremony; read-only discovery tools pass the classification gate untouched, so gathering the evidence to prove a tier down never requires authority; the expensive LLM layers run only on the slices of work that earned them. The system is heavy exactly where impact is heavy, and invisible where it is not. This is not UX polish — it is what makes the enforcement *survivable*, which is a precondition for it existing at all.

## 5. Justification and limits

The architecture follows one epistemic rule: **enforce what you can verify; advise what you cannot.** Path roles, exact payloads, fingerprints, tool results, observed text, and ledger state are runtime-observable, so they carry hard enforcement. Reasoning quality is not directly observable, so it is shaped — by taught discipline, by stream-time interrupts, by graded review — and the shaping is backed by consequences denominated in authority rather than by trust in self-report.

The honest account of where verification runs out:

**Form is checkable; substance is gameable — so the system anchors form to reality.** Any shape check can be satisfied by hollow content; this was a documented finding, not a hypothesis. The countermeasures all work by welding the checked form to facts the model does not control: exact-payload pre-commitment makes the *plan* real (a vague plan cannot be fingerprinted), effect-mismatch detection makes the *execution* real (deviation evaporates authority), tool-log cross-checking makes *evidence citations* real (a reference to something never read clears nothing), and semantic grading makes the *chain* itself reviewable by an adversarial reader that cannot be sweet-talked into granting anything.

Gaming remains possible; it just stops being cheaper than reasoning.

**Substance floors do not exist.** The deterministic layers can prove what a change *is* (comment-only, exports-unchanged) but not what it *means*; a semantically catastrophic change that is syntactically modest relies on path-role floors, the prosecutor's suspicion, and the grader's coherence review — all softer than a certificate. The framework knows the difference between "proved safe" and "nothing proved dangerous," and tiers only ever step down on the former; but the latter is where residual risk lives.

**Graderless degradation is real.** The semantic backstop requires a configured model. Without one — or on timeout — the system degrades to its deterministic skeleton by design (I4): shape checks, floors, certificates, fingerprints, leases, ledger. That skeleton is the enforcement majority, but the hollow-pass vector reopens at the margin when no grader is reachable.

**The lightest tier still leans on prompt pressure.** A `light` answer obligation is satisfied by a visible TARGET/DELTA/NEXT micro-pass — a shape check. Below the grading threshold, the framework is closer to structured guidance than to enforcement, accepted deliberately because the frictionless floor (§4h) outranks completeness at the low-impact end. The system's claim is not "no bad reasoning escapes"; it is "bad reasoning cannot cheaply acquire authority over things that matter."

## 6. Case study: the OMP binding

This repository binds the framework to [OMP](https://omp.sh)'s extension surface. It is one binding, not the framework itself — every mechanism below is an implementation of an abstraction from §3–§4, and the table is the map:

| Abstract mechanism | OMP surface | Code |
|---|---|---|
| Loop teaching (§3) | `HOLMES_SYSTEM_PROMPT` appended on every agent start; HALT/ENVISION/LOCATE/DELTA/CLASSIFY redirect; `/holmes`, `/holmes-goal`, `/holmes-status` commands; the playbook skill | `src/prompts.ts`, `rules/RULES.md`, `commands/`, `skills/holmes/SKILL.md` |
| Stream-time interruption (§2) | 8 TTSR rules (1 always-apply + 7 conditional) that abort generation mid-stream when a failure pattern forms — forward-chain intent, assumption-to-action, primitive-chaining plans, edit-without-verify, eval-as-mutation-bypass | `rules/` |
| Classification authority (§4a–b) | Extension-owned `holmes_classify` tool; prove-down Tier 4→1; returned tier/requirements/scope binding | `src/classification.ts` (`registerHolmesClassifyTool`) |
| Objective floors (§4c L1) | `detectObjectiveImpactFloors` — path roles, tool classes, payload syntax, ledger state; prose keywords demoted to `collectLexicalRiskHints` | `src/classification.ts` |
| Evidence certificates (§4c L2) | `computeEvidenceCertificates` — 12 kinds, from `whitespace_only`/`comment_only`/`ast_equivalent`/`exports_unchanged` through non-code kinds (`source_material_read`, `factual_cross_reference`, `coordination_plan_bounded`, `session_scoped_only`) | `src/classification.ts`, `src/types.ts` |
| Risk prosecutor (§4c L3) | `createExtensionOwnedRiskProsecutor` — quarantined packet, upward-only floor mapping, failure-inert | `src/classification.ts` |
| Pre-commitment (§4d) | `structuredEffect.exactPatch` / `exactContent` / `exactOps`, `exactOpaqueInput`; extension-computed effect fingerprints (`plannedActionEffectFingerprint`) | `src/classification.ts`, `src/types.ts` |
| Leases + ledger (§4e) | `MutationLease` (tools/paths/operation classes/effect fingerprints/mutation budget/file-state fingerprints/expiry); per-request ledger with scoped, supersedable floor entries and verification-failure floors | `src/types.ts`, `src/classification.ts` |
| Runtime gates (§4b) | `tool_call` interception in order: primitive burst → delegation → classification; distinct block messages each naming a safe next action | `src/main.ts`, `src/guards.ts` |
| Answer obligations (§4f) | `triageAnswerObligation` at `context` time (none: <200 chars, no code fence, ≤1 question, no reasoning verbs, no multi-part markers; full on reasoning-verb density); monotone `escalateAnswerObligation` (light at 600 visible chars / 1 code block / 4 tool calls; full at 8 tool calls / 3,000 chars + 2 code blocks / live Tier 3–4 record); one `nextTurn` demand, then `soft_accept` (`MAX_ANSWER_RETRIES = 1`) | `src/answer.ts`, `src/main.ts` |
| Checkpoint (§4f) | `holmes_checkpoint` tool — read-only, gate-exempt; target/chain/unknowns/plan; evidence and closure citations cross-checked against the request's tool log | `src/answer.ts` (`executeHolmesCheckpoint`) |
| Reasoning grader (§4f) | `createExtensionOwnedReasoningGrader` — `chain`/`closure`/`plan` axes, `coherent`/`hollow`/`incoherent` verdicts, hollow-flag cap 1, ≤2 calls per request, 5 s default / 8 s max timeout; mutation-path grading behind the `holmes-grade-mutation-passes` flag, default off | `src/grader.ts` |
| Verification pressure (§3 Establish) | Verify reminders appended to mutating tool results; verification outcomes feed the ledger | `src/main.ts`, `src/observation.ts` |

Two details show the invariants under load. The answer-gate state machine (`idle → obligated → awaiting_repair → satisfied | soft_accept`) is implemented exactly as the termination proof requires: digest-keyed state, capped counters, absorbing terminals — and `agent_end` deliberately awaits in-flight grader work before deciding, because OMP can deliver `agent_end` while `message_end` is still grading (a real race found in testing). And the checkpoint tool is exempted from the system's own gates at the top of the `tool_call` handler, because a blockable exit would violate I8.

**Binding the framework to another harness** requires four capabilities, none exotic:

1. **Observation of model output** — streamed or final visible text, distinguishable from hidden thinking, to detect visible passes and redact self-authorizing markers.
2. **Tool-call interception** — see every call with its arguments before execution, and block with a message.
3. **Message injection** — append context to a request, and schedule one continuation after the agent would otherwise stop, for obligation notices and the single checkpoint demand.
4. **Per-request state** — detect the request boundary (a digest of the latest user message suffices) to key obligations, leases, ledger, and caches.

Notably absent from the list: a final-answer veto (a delivered answer is immutable; the demand loop is post-hoc and forward-looking by design), forced tool choice (the checkpoint is demanded in prose, never compelled), and provider cooperation of any kind. The framework was designed against a minimal harness contract — anything it cannot get, it compensates for with bounded, named-exit demands rather than assuming a stronger runtime.

That is the framework: a backward-reasoning discipline a model could follow but predictably will not, wrapped in an enforcement architecture that never trusts the model's account of itself, prices authority in evidence, and is engineered — caps, exits, floors of zero friction — to remain installed. Prompts ask. HOLMES compels.
